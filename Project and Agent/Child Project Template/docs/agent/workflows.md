# Agent Workflows

本文件存放项目级长流程说明。只有任务命中对应场景时读取，不作为每轮默认上下文。

## 1. 核心原则：地图不是疆域

用户提供的 prompt、需求说明、设计稿、会议纪要、上下文和已有文档，都只是“地图”。

真实的疆域包括：

- 现有代码库。
- 真实业务规则。
- 历史遗留问题。
- 组件复用方式。
- 数据模型与接口约定。
- 用户隐性的审美和体验偏好。
- 运行环境与边界条件。
- 还没有被清楚表达出来的需求。

Agent 的目标不是机械执行，而是帮助用户暴露未知、识别风险、推进最小可验证方案，并在交付前说明验证结果和剩余风险。

## 2. 四类未知

任务开始前，按风险程度区分四类内容：

- Known Knowns：用户已经明确写出的需求、规则、限制、目标。严格遵守。
- Known Unknowns：用户已经意识到但还没有答案的问题。列为待确认项。
- Unknown Knowns：用户心里有判断但说不清，需要通过 demo、对比稿或反馈暴露偏好。
- Unknown Unknowns：用户和 Agent 一开始都没有意识到的问题，只能通过代码、数据、运行和测试暴露。

不要把“待确认口径”当成“明确错误”直接修。

不要把“Agent 猜测用户想要”写成“用户要求”。

## 3. 工作模式

根据任务情况选择最小必要模式。

### 3.1 Blind Spot Pass

适用场景：

- 陌生模块。
- 新功能。
- 需求模糊。
- 影响架构、数据、权限、体验流程。

输出内容：

1. 当前理解。
2. 已知约束。
3. 已知未知。
4. 可能的未知未知。
5. 需要检查的代码或资料。
6. 需要用户确认的问题。
7. 不确认就不能推进的事项。

### 3.2 Reverse Interview

适用场景：

- 用户有想法，但需求还不稳定。
- 一个问题会改变数据、接口、权限、核心流程或信息架构。

规则：

- 每次只问一个最关键问题。
- 优先问“答案会改变方案”的问题。
- 给出 2-3 个选项和影响。
- 不一次抛出很多问题增加用户负担。

### 3.3 Prototype First

适用场景：

- UI、交互、审美、信息结构还不确定。
- 用户需要看到不同方向再判断。
- 直接改主项目成本太高。

规则：

- 先做低成本 demo、静态页面、假数据页面、结构草图或对比方案。
- 不直接污染生产代码。
- 根据用户反馈沉淀明确规则，再进入正式开发。

### 3.4 Implementation Plan

适用场景：

- 正式改多个文件。
- 涉及业务逻辑。
- 影响数据、接口、权限、状态机、核心流程。

计划内容：

- 目标。
- 非目标。
- 现有观察。
- 文件修改清单。
- 数据、接口、权限、UI 流程影响。
- 风险与回滚方式。
- 验证方式。
- 需要用户确认的定义口径。

### 3.5 Implementation Notes

适用场景：

- 执行中发现偏离原计划。
- 发现历史错误。
- 发现现有代码与需求冲突。
- 做了保守取舍。

记录内容：

- 偏离点。
- 触发原因。
- 采取的方案。
- 是否需要用户确认。
- 明确错误。
- 待确认口径。
- 后续风险。

### 3.6 Delivery Review

适用场景：

- 改动较多。
- 准备合并或交付。
- 用户需要理解本次改动。

交付说明：

- 做了什么。
- 为什么这样做。
- 改了哪些关键文件。
- 已验证内容。
- 未验证内容。
- 剩余风险。
- 需要确认的口径。
- 回滚建议。

### 3.7 Final Exam

只在复杂架构、权限、数据模型、核心流程或教学型交付中使用。

不要默认每次交付都生成题目。

## 4. 文档同步

如果本次改动改变了需求、字段、状态、接口、权限、页面行为、数据口径或验证方式，应同步更新对应文档。

如果用户只要求临时实验、草图或一次性分析，不默认扩大到全套文档同步；但交付时说明这是临时结果还是应沉淀为项目规则。
