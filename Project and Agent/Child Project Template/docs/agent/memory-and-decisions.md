# Memory and Decisions

本文件存放项目级主动沉淀、决策记录和口径判断规则。只有任务涉及沉淀、决策、重复错误或项目规则演化时读取。

## 1. 明确错误 vs 待确认口径

明确错误：

- 与已确认需求冲突。
- 与代码实际运行结果冲突。
- 与测试结果冲突。
- 与项目内已有规范冲突。
- 与数据证据冲突。

待确认口径：

- 存在多个合理解释。
- 不同解释会导致不同实现。
- 项目内没有明确规范。
- 用户可能有隐性偏好。
- Agent 无法通过代码事实自行判断。

处理规则：

- 明确错误可以提出修复建议。
- 如果修复会影响数据模型、接口、权限、历史兼容或核心流程，仍需先确认。
- 待确认口径必须给出选项、影响和建议，不擅自拍板。

## 2. 主动沉淀机制

Agent 在项目对话过程中，应主动识别可沉淀内容，而不是等待用户专门要求。

用户可以随时要求：

- 本轮不沉淀。
- 暂停沉淀。
- 只列候选，不写入文件。
- 只更新指定文件。

## 3. 默认识别的沉淀候选

以下内容应被识别为沉淀候选：

- 用户明确说“以后都这样”“这个项目默认这样”“这个口径记住”。
- 本项目反复出现的固定术语、字段、状态、命名、目录、验证命令。
- 已通过代码、数据、运行结果或用户确认支持的项目规则。
- 已踩过且后续容易重复的错误、失败原因和修复方式。
- 影响后续任务判断的产品口径、业务边界、交互偏好或技术约束。
- 当前文档与代码之间已经确认的偏差。

## 4. 写入位置

- 项目执行规则、固定读序、固定验证命令、输出风格：写入项目 `AGENTS.md`。
- 已确认项目经验、业务口径、命名规则、后续复用事实：写入 `.learnings/LEARNINGS.md`。
- 已踩错误、失败原因、修复方式、避免方式：写入 `.learnings/ERRORS.md`。
- 用户正式确认的长期项目决策：写入 `docs/decision-log.md`，如文件不存在且确有需要时再创建。
- 跨项目都适用的长期规则：先提示用户确认，再考虑更新全局 `/Users/guofu/.codex/AGENTS.md`。

## 5. 默认可直接写入

满足以下条件时，Agent 可以在完成任务时直接写入 `.learnings/`：

- 内容只影响本项目。
- 有明确证据支持。
- 不包含敏感信息。
- 不改变业务决策、接口合约、权限模型或核心流程。
- 写入后不会覆盖用户已有内容，只做追加或局部补充。

## 6. 需要先确认再写入

以下内容不能擅自写入 `AGENTS.md` 或正式决策文档：

- 用户偏好不够明确，只是 Agent 推测。
- 会改变后续默认工作方式。
- 会影响数据模型、接口、权限、部署、核心流程。
- 会把一次性任务经验升级成长期规则。
- 会影响多个项目或全局规则。

## 7. 交付前沉淀检查

每次任务结束前，Agent 应快速检查是否有可沉淀内容。

如果已经写入，应在最终回复说明：

- 写入了哪里。
- 写入了什么类型的内容。
- 为什么值得沉淀。

如果发现候选但不适合直接写入，应在最终回复列为“沉淀候选”，请用户确认是否写入。

## 8. `.learnings/LEARNINGS.md` 建议格式

```md
## 规则或经验标题

- 类型：业务口径 / 命名规则 / 验证方式 / 体验偏好 / 技术约束
- 证据：来自用户确认 / 代码事实 / 测试结果 / 数据结果
- 内容：
- 适用范围：
- 记录时间：
```

## 9. `.learnings/ERRORS.md` 建议格式

```md
## 错误标题

- 现象：
- 原因：
- 修复：
- 下次避免方式：
- 证据：
- 记录时间：
```
