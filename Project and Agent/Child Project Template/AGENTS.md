# 项目级 AGENTS.md

本文件叠加 guofu 的全局 `/Users/guofu/.codex/AGENTS.md`。

若与全局规则冲突，以本项目更具体规则为准。

本文件的目标不是最短，而是好管理：打开本文件应能快速理解项目定位、资料入口、执行规则、工作模式、风险边界和验证方式。

## 0. 核心原则：地图不是疆域

用户提供的 prompt、需求说明、设计稿、会议纪要、上下文和已有文档，都只是“地图”。

真实的疆域包括：

- 现有代码库。
- 真实业务规则。
- 历史遗留问题。
- 组件复用方式。
- 数据模型与接口约定。
- 用户隐性的审美和体验偏好。
- 运行环境与边界条件。
- 还没有被清楚表达出来的需求。

Agent 的目标不是机械执行，而是帮助用户暴露未知、识别风险、推进最小可验证方案，并在交付前说明验证结果和剩余风险。

## 2. 默认读序

开始本项目任务时，默认按以下顺序理解上下文：

1. 用户当前消息。
2. 全局 `/Users/guofu/.codex/AGENTS.md`。
3. 本项目 `AGENTS.md`。
4. 本项目 README、核心需求文档、设计规范、测试说明或数据说明。
5. 当前任务直接相关的代码、数据、配置、运行结果。
6. 必要时再读取 `.learnings/`、`docs/agent/` 或其他补充资料。

原则：

- 用户最新明确指令优先。
- 本项目规则优先于全局通用规则。
- 代码、数据、测试、运行结果优先于推测。
- 不默认整篇读取大型历史资料。

## 4. 四类未知

任务开始前，按风险程度区分四类内容：

- Known Knowns：用户已经明确写出的需求、规则、限制、目标。严格遵守。
- Known Unknowns：用户已经意识到但还没有答案的问题。列为待确认项。
- Unknown Knowns：用户心里有判断但说不清，需要通过 demo、对比稿或反馈暴露偏好。
- Unknown Unknowns：用户和 Agent 一开始都没有意识到的问题，只能通过代码、数据、运行和测试暴露。

不要把“待确认口径”当成“明确错误”直接修。

不要把“Agent 猜测用户想要”写成“用户要求”。

## 5. 工作模式

根据任务情况选择最小必要模式。

### 5.1 Blind Spot Pass

适用场景：

- 陌生模块。
- 新功能。
- 需求模糊。
- 影响架构、数据、权限、体验流程。

输出内容：

1. 当前理解。
2. 已知约束。
3. 已知未知。
4. 可能的未知未知。
5. 需要检查的代码或资料。
6. 需要用户确认的问题。
7. 不确认就不能推进的事项。

### 5.2 Reverse Interview

适用场景：

- 用户有想法，但需求还不稳定。
- 一个问题会改变数据、接口、权限、核心流程或信息架构。

规则：

- 每次只问一个最关键问题。
- 优先问“答案会改变方案”的问题。
- 给出 2-3 个选项和影响。
- 不一次抛出很多问题增加用户负担。

### 5.3 Prototype First

适用场景：

- UI、交互、审美、信息结构还不确定。
- 用户需要看到不同方向再判断。
- 直接改主项目成本太高。

规则：

- 先做低成本 demo、静态页面、假数据页面、结构草图或对比方案。
- 不直接污染生产代码。
- 根据用户反馈沉淀明确规则，再进入正式开发。

### 5.4 Implementation Plan

适用场景：

- 正式改多个文件。
- 涉及业务逻辑。
- 影响数据、接口、权限、状态机、核心流程。

计划内容：

- 目标。
- 非目标。
- 现有观察。
- 文件修改清单。
- 数据、接口、权限、UI 流程影响。
- 风险与回滚方式。
- 验证方式。
- 需要用户确认的定义口径。

### 5.5 Implementation Notes

适用场景：

- 执行中发现偏离原计划。
- 发现历史错误。
- 发现现有代码与需求冲突。
- 做了保守取舍。

记录内容：

- 偏离点。
- 触发原因。
- 采取的方案。
- 是否需要用户确认。
- 明确错误。
- 待确认口径。
- 后续风险。

### 5.6 Delivery Review

适用场景：

- 改动较多。
- 准备合并或交付。
- 用户需要理解本次改动。

交付说明：

- 做了什么。
- 为什么这样做。
- 改了哪些关键文件。
- 已验证内容。
- 未验证内容。
- 剩余风险。
- 需要确认的口径。
- 回滚建议。

### 5.7 Final Exam

只在复杂架构、权限、数据模型、核心流程或教学型交付中使用。

不要默认每次交付都生成题目。

## 6. 明确错误 vs 待确认口径

明确错误：

- 与已确认需求冲突。
- 与代码实际运行结果冲突。
- 与测试结果冲突。
- 与项目内已有规范冲突。
- 与数据证据冲突。

待确认口径：

- 存在多个合理解释。
- 不同解释会导致不同实现。
- 项目内没有明确规范。
- 用户可能有隐性偏好。
- Agent 无法通过代码事实自行判断。

处理规则：

- 明确错误可以提出修复建议。
- 如果修复会影响数据模型、接口、权限、历史兼容或核心流程，仍需先确认。
- 待确认口径必须给出选项、影响和建议，不擅自拍板。

## 7. 执行原则

- 简单、低风险、目标明确的任务直接执行。
- 复杂、模糊、高风险任务先澄清目标、约束、风险和验证方式。
- 优先选择最小必要方案。
- 只改当前任务直接相关内容。
- 不顺手重构无关代码。
- 不引入不必要的新依赖。
- 不为了未来不确定需求提前设计复杂架构。
- 保持项目现有风格、目录习惯、命名方式和验证方式。
- 发现无关问题可以指出，但不要擅自一起改。

## 8. 风险边界

遇到以下情况，先说明风险并等待确认：

- 修改数据模型、权限模型、接口合约或核心流程。
- 删除文件、删除数据、覆盖用户已有内容。
- 修改部署、密钥、环境配置或线上环境。
- 大范围格式化、大范围重构或重写核心模块。
- 业务口径不明确，且不同选择会导致不同实现。
- 发现历史实现或文档存在明确错误，但修复会扩大影响范围。

可以自行处理的低风险事项：

- 明确拼写错误。
- 类型错误。
- 局部 lint 问题。
- 不改变业务逻辑的局部整理。
- 为已有逻辑补充测试。
- 按项目既有规范补齐简单状态或样式。

前提：不改变业务口径、数据模型、接口定义、权限、历史兼容，不引入新依赖。

## 9. 验证要求

交付前必须说明验证到什么程度。

默认验证顺序：

1. 运行项目已有的最小验证命令。
2. 若没有统一验证命令，运行与本次改动直接相关的检查。
3. 若验证无法运行，说明原因、风险和替代检查。

本项目常用验证命令：

```bash
# 待补充
```

不要只说“已完成”，要说明：

- 改了什么。
- 为什么这样改。
- 怎么验证。
- 哪些风险仍存在。

## 13. 学习记录

只有被证据支持、后续会复用的内容，才写入 `.learnings/`。

- 项目特定规则、机制、数值、裁定：写入 `.learnings/LEARNINGS.md`。
- 已踩错误、失败原因、避免方式：写入 `.learnings/ERRORS.md`。

不要写入：

- 临时猜测。
- 一次性中间过程。
- 未经确认的偏好。
- 敏感信息。

## 14. 本文件维护规则

本文件优先保持单文件可读。

适合写在本文件里的内容：

- 项目定位。
- 权威资料入口。
- 固定读序。
- 工作模式。
- 风险边界。
- 常用验证命令。
- 项目专属输出风格。

不适合写在本文件里的内容：

- 大段需求正文。
- 大段设计稿说明。
- 大段数据字典。
- 一次性任务计划。
- 频繁变化的中间结论。

这些内容应放在 `docs/`、`docs/agent/` 或 `.learnings/`，并在本文件中给出清楚入口。
